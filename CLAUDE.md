# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
npm run dev         # Vite dev server on :5173 (host=true, reachable from LAN)
npm run build       # tsc -b && vite build → dist/ (PWA service worker generated by vite-plugin-pwa)
npm run typecheck   # tsc -b --noEmit (only type-checking tool; no ESLint/tests configured)
npm run preview     # serve built dist/
npm run deploy      # build + wrangler pages deploy dist --project-name=card-stitcher
```

There is no test runner and no linter. `npm run typecheck` is the only CI-style check.

## Architecture

Card Stitcher is a **100% client-side PWA**. There is no backend. All images, metadata, and LLM API keys stay in the browser. See `PRD.MD` for full product context; `DEPLOY.md` for Cloudflare Pages setup.

### Data model and persistence

- Domain types live in `src/types/card.ts`: `Card` (id, pageOrder, pages, metadata, coverBlobKey) and `PageAsset` (with `originalBlobKey` / `enhancedBlobKey` / `useEnhanced` so the enhance flow is non-destructive). `MAX_PAGES = 8`.
- `src/lib/db.ts` wraps `idb-keyval` with two key namespaces: `card:{id}` → JSON card records, `blob:{key}` → image Blobs. Deleting a card iterates `pageOrder` and removes each blob. `requestPersistence()` asks the browser to make storage durable (called from `EditorPage`).
- Blob URLs are created via `blobUrl(key, blob)` in `src/lib/imageUtils.ts` and cached by key so React re-renders don't leak URLs.

### State (zustand)

Three stores, each responsible for one concern:
- `useCardStore` — library list (read-only view of saved cards).
- `useEditorStore` — the in-progress card being edited: `pageOrder`, `pages`, `pageUrls`, `metadata`, `dirty` flag. `hydrate(id)` loads from IDB, `persist()` writes back. The enhance lifecycle (`applyEnhance` / `toggleEnhanced` / `discardEnhanced`) swaps `blobKey` between original and enhanced while keeping both in IDB until the user discards.
- `useEnhanceStore` — enhance UI state.

### Routes

`src/App.tsx` defines three routes: `/` (LibraryPage), `/edit/:cardId` (EditorPage), `/view/:cardId` (ViewerPage). Client-side SPA — `public/_redirects` rewrites every path to `index.html` on Cloudflare Pages.

### LLM autofill (Bring-Your-Own-Key)

- Entry point: `src/lib/ai.ts`. Selected provider + per-provider API key + per-provider model are stored in `localStorage` under `cs:provider`, `cs:{provider}-key`, `cs:{provider}-model`.
- Provider adapters live in `src/lib/providers/{anthropic,openai,gemini}.ts` and are **dynamically imported** so the SDK for the unused providers doesn't land in the initial bundle.
- Shared contract in `src/lib/providers/types.ts`: every adapter takes `(apiKey, model, AutofillInput)` and returns `Partial<CardMetadata>`. The `MetadataSchema` (zod) + `SYSTEM_PROMPT` + `toMetadataPatch()` are shared by all adapters — changes to the prompt or schema go here, not per-provider.
- `DEFAULT_MODEL` in that file pins the default model per provider. When swapping models, update both this map and any model-specific request options in the adapter.

### Enhance pipeline (dewarp + background removal)

Lives in `src/lib/enhance/`. Two pluggable backends behind `EnhanceBackend`:
- `classical.ts` — OpenCV.js. The main thread is kept free: all CV work happens in `cv-worker.ts` (classic Web Worker, **not** a module worker — module-worker + dynamic import hung OpenCV init; see comment at the top of `cv-worker.ts`). OpenCV is loaded via `importScripts` from `docs.opencv.org` at runtime. `cv-client.ts` is the singleton main-thread RPC client.
- `onnx.ts` — onnxruntime-web, ~45 MB model downloaded on first use and cached by the service worker. Still runs fully on device.

Backend selection is persisted in `localStorage` under `cs:enhance-backend`. Use `enhancePage(source, opts, backendId?)` from `src/lib/enhance/index.ts`; backends are dynamically imported. The service-worker runtime-caching rules in `vite.config.ts` (`opencv-cdn-v1`, `ort-runtime-v1`, `enhance-models-v1`) make the first-run downloads permanent — **do not move or rename those URL patterns casually**, you'll invalidate existing users' caches.

### Exporters

`src/components/export/exporters/` — one file per format (`pdfExporter` via pdf-lib, `gifExporter` via gif.js, `zipExporter` via jszip). All run in the browser from the already-loaded blobs.

### PWA / build config

- `vite.config.ts` configures `vite-plugin-pwa` with `registerType: 'autoUpdate'`, workbox `skipWaiting` + `clientsClaim`, and explicit `globIgnores` for `**/*.onnx` and `**/assets/opencv-*.js` (too large to precache — they use runtime caching instead). `maximumFileSizeToCacheInBytes` is 12 MB.
- `worker: { format: 'iife' }` is required for the OpenCV classic worker.
- `public/_headers` sets long-lived immutable cache for `/assets/*` and forces `sw.js` to revalidate. Don't add cache-busting to `sw.js`.

## Conventions worth knowing

- Provider and enhance-backend modules are code-split via dynamic `import()`. Keep it that way: importing `@anthropic-ai/sdk`, `@google/genai`, `openai`, `onnxruntime-web`, or `@techstark/opencv-js` at the top level of any always-loaded module will bloat the initial bundle.
- Enhance results must preserve `originalBlobKey` so users can toggle/discard — never overwrite the original blob in IDB.
- `DEFAULT_MODEL.anthropic` is `claude-opus-4-7` and `DEFAULT_MODEL.gemini` is `gemini-2.5-flash`; confirm model IDs before bumping.
